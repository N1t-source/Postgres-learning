Hands-on work

Lab index

Each lab is deliberately small. The goal is to prove the concept, not to ship a feature.

SessionLessonLab objective
01Why databases existCompare a flat country list with a tiny relational model and explain which one better supports change.
02Tables, rows, columns, and schemasSketch a countries table and label which facts belong in rows versus columns.
03Primary keys and row identityDesign a countries table that can survive duplicate country names or renamed records.
04Data types and NULLAdd typed columns to a country table and identify which columns should allow NULL.
05What PostgreSQL is and how psql fitsPractice the connect-inspect-query loop with a tiny local database.
06SELECT, FROM, and aliasesWrite a query that reads country names and continent names from a simple table.
07WHERE and comparison operatorsFind countries from one continent and explain why the other rows are excluded.
08ORDER BY and LIMITShow the largest countries first and the smallest result set needed for a report.
09INSERTAdd a few countries to the table and confirm the rows are really there.
10UPDATE and DELETEUpdate one country fact and delete one fake row using a narrow WHERE clause.
11Creating a database and inspecting itCreate the learning database and inspect it with psql commands.
12Creating tables and describing themCreate a countries table and inspect it with describe commands.
13text, varchar, integer, and numericRefine the country table so columns use a type that matches the data.
14boolean, date, time, and timestampStore a few country facts such as membership status or independence dates.
15serial vs identity and safe droppingCreate a table that auto-generates ids and then practice safe dropping in a sandbox.
16Entities and attributes from country requirementsDraft the first country model with countries as the core entity.
17Regions and continentsCreate region and continent reference tables and connect them to countries later.
18One-to-many relationshipsAdd a child table that points back to countries.
19Many-to-many relationships and join tablesModel languages spoken in multiple countries through a bridge table.
20Normalization and naming conventionsReview the country model and identify one repeated fact that should move elsewhere.
21PRIMARY KEY and NOT NULLAdd constraints to the countries table and try an invalid insert.
22UNIQUE, DEFAULT, and CHECKAdd a uniqueness rule for country codes and a check for a valid continent list.
23Foreign keys and referential actionsConnect cities to countries and test what happens when a parent is deleted.
24Integrity failure casesTry a few bad inserts or deletes and observe the failure messages.
25INNER JOINShow countries alongside the cities that belong to them.
26LEFT JOINList countries even when no city rows exist yet.
27RIGHT JOIN and FULL OUTER JOINCompare join direction and see how the row set changes.
28CROSS JOIN and self joinsBuild combinations of continents and years, then compare two cities from the same table.
29COUNT and SUMCount countries and sum a simple population column.
30AVG, MIN, and MAXReport the smallest, largest, and average population for the sample countries.
31GROUP BYCount countries by continent and compare the groups.
32HAVING and summary reportsShow only continents with more than one country in the sample data.
33Seed files and fake dataWrite a seed file for a small set of countries and regions.
34CSV import and reset scriptsImport the mock countries file and rebuild the database from scratch.
35Testing inserts and constraintsAdd a few countries to the table and confirm the rows are really there.
36Testing joins and query resultsCheck join results for countries with cities and countries without cities.
37BEGIN, COMMIT, and ROLLBACKWrap a few changes in a transaction and reverse them once.
38Atomicity and consistencyBreak a transaction on purpose and inspect the database afterward.
39Indexes and B-tree basicsCreate an index on a lookup column and compare the idea to a full table scan.
40EXPLAIN, EXPLAIN ANALYZE, and query planningInspect a simple query plan and compare it to the table's shape.
