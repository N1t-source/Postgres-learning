Big Idea

Design a table from scratch with sensible columns and types.

When you work through this topic, do not think of it as a single trick. Think of it as part of how a database team reasons about data shape, correctness, and readable queries.

For this lesson, the thing to keep repeating is: CREATE TABLE, defaults, identity columns.

Worked Walkthrough

  1. Look at the problem first, not the syntax.
  2. Identify the data object you are touching, such as a table, row, filter, or relationship.
  3. Write the smallest possible SQL or database action that proves the idea.
  4. Inspect the result before you move on, even if it looks obvious.

SQL Examples

CREATE TABLE cities (id SERIAL PRIMARY KEY, name TEXT NOT NULL, country_id INT);
CREATE TABLE regions (id SERIAL PRIMARY KEY, name TEXT UNIQUE);

These are starter examples. Change the table name, filter, or output column and predict what happens before you run them.

Common Mistakes

  • Using the right syntax but the wrong mental model.
  • Changing too many things at once and losing track of what changed the result.
  • Skipping the check step and assuming the first answer is correct.

Teaching Notes

Say the concept out loud in plain English before you touch the keyboard.

Ask yourself which part is the data, which part is the rule, and which part is the result.

If the lesson feels easy, slow down and explain why it works. That is where the understanding lives.

Lab

Lab objective: Create a cities table with keys and timestamps

This is the hands-on checkpoint for the session. Work in small steps, verify every result, and only move forward once the current screen or query output matches the lesson goal.

Lab Setup

  • Open psql or the lesson's approved client and connect to PostgreSQL.
  • Open the lesson database used in this session. If it does not exist yet, create it first from the lesson instructions.
  • Keep the lesson notes visible so you can compare each step with the prompt.

Starter SQL

If you are new to the lesson object, this is the shape of a table creation step:

CREATE TABLE lesson_notes (
  id SERIAL PRIMARY KEY,
  title TEXT NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()

The exact table name changes by session. The point is to see how a table is defined before you query or modify it.

Step-by-Step

  1. Read the objective out loud and restate it in plain English.
  2. Find the table, view, query, or database object that this lesson is about.
  3. Complete the core action for the lab: Create a cities table with keys and timestamps.
  4. Run one small variation and predict the result before you execute it.
  5. Write down anything unexpected, even if the query still worked.
  6. Explain the result to yourself before you leave the lab section.

Success Criteria

  • You can describe what changed after the lab action.
  • You can point to the exact SQL or UI action that proved the concept.
  • You can predict a one-step variation before trying it.

Common Mistakes

  • Skipping the setup step and changing too many things at once.
  • Reading the answer before checking the output yourself.
  • Not naming the specific table, column, or database object involved.

Extra Practice
